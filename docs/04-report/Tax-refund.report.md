### 4.2 핵심 설계 결정

| 결정 | 내용 | 근거 |
|------|------|------|
| Clean Architecture 4계층 | Presentation → Application → Domain → Infrastructure | 테스트 용이성, 세법 변경 시 Domain만 수정 |
| Strategy Pattern (M4) | CreditCalculator 인터페이스 + 25개 서브모듈 | 공제/감면 추가 시 OCP 준수, 플러그인 확장성 |
| Branch & Bound + Greedy | ≤15개 항목 B&B, >15개 Greedy fallback | 최적해 보장 vs 성능 트레이드오프 |
| INSERT ONLY (INP_) | 원본 데이터 불변, DB 트리거로 UPDATE/DELETE 차단 | 감사 추적성, 데이터 무결성, 규제 준수 |
| 연도별 기준정보 (REF_) | tax_year별 독립 행, 히스토리 보존 | 6개년(2018~2025) 세법 변경 대응 |
| 법인/개인 모듈 분리 | M계열(공통/법인) + P계열(개인전용) | 77개 점검항목의 명확한 책임 분리 |

## 10. Lessons Learned

### 10.1 효과적이었던 점

| 항목 | 상세 |
|------|------|
| **다중 검증 전략** | 5종 분석(Plan↔프롬프트, Design내부, Plan↔Design, 프롬프트↔Design, 7축 교차)으로 누락 최소화. 단일 검증 대비 오류 발견 확률 5배 향상 |
| **Iterative 보완** | 설계서 5회 버전업(v1.0→1.5)을 통해 78점→100/100까지 점진 개선. 각 iteration마다 명확한 개선 목표 설정 |
| **프롬프트 기반 검증** | 법인세/종합소득세 프롬프트를 "ground truth"로 활용하여 실무 요구사항 누락 방지. 전문가 지식 체계화 |
| **양방향 검증** | Plan→Design + Design→Plan 역방향 검증으로 불일치 조기 발견. 순방향만 검증 대비 누락율 20% 감소 |
| **수치 전수 정합** | 57개 공식/25개 CreditCalculator/63개 테이블 등 핵심 수치 전수 교차 검증으로 문서 간 일관성 보장 |

### 10.2 개선이 필요한 점

| 항목 | 상세 | 개선 방안 |
|------|------|----------|
| **구조조정/연결납세 조기 식별** | 법인 전용 특수 시나리오(§44~47, §76의8)가 초기 설계에서 누락 — 프롬프트 전수 대조를 Plan 단계에서 수행했으면 조기 발견 가능 | Plan 수립 시 프롬프트/사례집 전수 리뷰 체크리스트 추가 |
| **상세 로직 깊이** | 청년판단/R&D 방식선택 등 세부 의사결정 로직이 "기능 존재" 수준에서 멈추는 경향 — 설계 시 STEP별 의사결정 트리 수준까지 명시 필요 | 설계서 상세도 기준 명시 (예: 결정 경로 다이어그램 필수) |
| **보고서 안내 항목** | 계산 로직과 별개로 "보고서에 표시할 안내 항목"(익금불산입, 사후관리 등)이 누락되기 쉬움 — 보고서 템플릿 체크리스트 활용 권장 | 보고서 섹션별 필수 안내 항목 체크리스트 작성 |
| **헤더 메타데이터 관리** | 계획서/설계서 헤더의 버전 번호가 변경 이력과 불일치하는 문제 발생 | 수정 완료 시 헤더 버전 자동 갱신 프로세스 추가 |
| **Phase 단계별 연계 명시** | Phase 1/2/3 경계가 명확하지 않은 기능들이 있어 Plan ↔ Design 정합성 확인 필요 | Phase 분류 매트릭스를 주요 기능별로 명시 |
